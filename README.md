# grazioso-salvare-dashboard
CS 340 Project Two dashboard and portfolio artifacts (CRUD module + MongoDB integration)
Module Eight Journal  Reflection
Maintainable, readable, adaptable code (and the CRUD module)
For Project One and Project Two, the biggest thing that helped keep my code clean was separating the database logic from the dashboard logic. Instead of putting database calls everywhere in the UI code, I used a CRUD style Python module as the “middle layer” between the widgets and MongoDB. The advantage of doing it this way was that it kept each piece focused: the dashboard handled display and user interaction, and the CRUD module handled create/read/update/delete operations. It also made debugging easier, because if something broke, I could usually tell whether it was a UI problem or a database problem. In the future, I can reuse the CRUD module as a starting point for any project that needs database access—especially if I build another dashboard or API. I’d only need to swap collections, fields, and validation rules instead of rewriting the database connection every time.
How I approach problems as a computer scientist
I approached the Grazioso Salvare requirements by breaking them down into smaller pieces and testing each piece before moving on. I focused on the database first making sure the data was stored correctly and that queries returned what I expected because the dashboard can’t be reliable if the data layer is shaky. Once the database queries were working, I connected the dashboard widgets one by one and tested them with real data to make sure the UI matched the requirements. This felt different from some past assignments because it was less about finishing a single script and more about building a working system with multiple parts that depend on each other. Going forward, I’d use the same strategy: confirm the data model first, validate sample queries, then build features in small testable steps so the final system stays stable as it grows.
What computer scientists do and why it matters
Computer scientists solve real problems by turning requirements into systems that are reliable, scalable, and usable. A project like this matters because it’s not just “writing code” it’s creating something that helps people make decisions faster and with fewer mistakes. For a company like Grazioso Salvare, having a working dashboard connected to a database means they can search, filter, and act on data quickly instead of manually digging through records. That saves time, reduces human error, and makes the organization more effective. This project also showed me how much value comes from building tools that make other people’s work easier, not just building something that runs.
